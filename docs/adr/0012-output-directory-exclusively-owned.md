# The output directory is exclusively gqlc's

`gqlc generate` treats the configured output directory as a wholly
generated artifact: every successful run replaces its contents
wholesale, and no file that gqlc did not generate may live there. This
is a deliberate rejection of sqlc's tolerant posture — where users drop
hand-written files (a `tx.go` with helper methods) beside the generated
ones — and it is guarded so that the destructive semantics can never
eat hand-written work.

## Context

- Go permits attaching methods to a type only from inside the type's
  package. Tolerating hand-written files in the output directory is
  what enables the sqlc-style same-package extension pattern; it is the
  only substantive argument for partial ownership.
- Partial ownership has a real failure mode: a renamed or deleted named
  query leaves its previously generated file behind — a phantom method
  that keeps compiling while drifting from the queries directory.
- Every generated file carries the canonical marker header
  (`// Code generated by gqlc <version>. DO NOT EDIT.`), so "did gqlc
  write this file?" is mechanically answerable.
- `codegen.Generate` is pure — it returns the full `[]File` set in
  memory. The CLI owns all filesystem writes, so writing can be ordered
  strictly after successful generation.

## Decision

The output directory contains only gqlc output. Each `generate` run
wipes and rewrites it — but the wipe is guarded by an **all-marked
tripwire**: it proceeds only when the directory is absent, empty, or
every existing file bears the generated-code marker. Otherwise the run
aborts, naming the offending files. gqlc never deletes a file it cannot
prove it generated.

Users extend generated code by wrapping the generated package from
their own package — embedding or delegating — never by adding files
alongside it.

## Considered options

- **Tolerate unmarked files, delete only stale marked ones**: keeps the
  sqlc extension pattern but leaves the directory only partially owned;
  rejected in favour of the cleaner boundary — extension by wrapping is
  the better-factored consumer posture anyway.
- **Unconditional wipe (`RemoveAll` + rewrite)**: simplest contract,
  but one config typo — `output: ./internal` instead of
  `./internal/db` — deletes an application source tree. The tripwire
  converts that scenario into an error message at the cost of one
  directory scan per run.

## Consequences

- No phantom generated files, ever: renames and deletions in the
  queries directory are fully reflected each run.
- `generate` can never destroy hand-written work; the worst outcome of
  a misconfigured output path is an abort that names the files it
  refused to touch.
- Because generation is pure, errors surface before any write: a
  failed run performs no wipe and leaves the directory untouched.
- Methods cannot be attached to generated types from user code; that is
  accepted, not accidental.
