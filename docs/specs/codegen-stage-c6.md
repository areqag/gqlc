# Stage C6 spec — codegen: polish (version stamp, session config, gqlc-0aa re-scope)

The implementation brief for Stage C6 of `internal/codegen`, extending
the merged C5 slice (`docs/specs/codegen-stage-c5.md`) with the polish
sweep ADR 0010 D7 places at C6: **version-stamp hardening** — pin the
exact stamp format, its source-of-truth for the version string, and
the reproducibility contract callers can rely on; **session-config
polish** — audit the C0 `SessionConfig{AccessMode: access}` shape and
the C0 `WithTx` seam against the axes C1–C5 lit up (per-method access,
managed-transaction chaining, cluster routing), pin the axes v1 locks
in and the ones a future config-effort widens; and **`gqlc-0aa`
re-scope** — the pre-C0 driver-wiring bead filed before ADR 0010 D4
killed the runtime package, closed out now that C0–C5 subsumed every
capability that would have lived in a runtime layer. Build this
**test-first**. Scope, sequencing, and error posture inherit from
ADR 0010 and the C0 / C1 / C2 / C3 / C4 / C5 specs unchanged; this
document revises only the sections C6 touches.

Stage C6 keeps the C5 file set (`db.go` / `querier.go` / `models.go`
/ `<name>.cypher.go`) byte-identical for the parts C6 does not touch,
introduces one new source file (`internal/codegen/version.go`, §4.1)
carrying the `version` package-level variable that `generate.go:22` currently owns,
tightens the emitted-file header contract (§5.2) so the byte-format
is pinned and the version substitution is the *only* per-invocation
variability, adds one new unit test asserting the header regex
(§6.1) — no new sentinels, no new fixtures, no new goldens. C6 does
not touch the `driverDB.run` seam (§5.3), does not widen the `New` /
`WithTx` shape (§5.4), does not add a `SessionConfig` option surface
(§5.4 grill). The C5 property → Go type mapping, the six-source
collision sweep, and the fifteen sentinels stand byte-identical; the
one sentinel touched is the per-stage rename `ErrOutOfC5Scope` →
`ErrOutOfC6Scope` (§7).

per-query derivation, no new admission gate, no new emission. The
sole compile-level change is the extraction of the `version` package-level variable

in (§5.4), the `gqlc-0aa` re-scope (§8) — that this spec pins so
future readers can see the closure decision.

### 2.1 The C6 kernel structure — unchanged

Phase Z / Phase A / Phase B / cross-query sweep run byte-identical to
C5. The kernel dispatch on `preparedRow.Kind` still enumerates the
eight `columnKind` values C5 pinned (`columnProperty`, `columnNode`,
`columnEdge`, `columnTemporal`, `columnScalar`, `columnList`,
`columnAny`, `columnEdgeUnion`). No new kind, no new arm.

### 2.2 Purity, determinism, short-circuit — unchanged, with one tightening

C5 §2.5's three invariants stand. The C6 tightening lives inside the
**deterministic** axis: the header substitution is the one text-level
variability across builds of the same binary, so the C0 double-run
test (`TestDoubleRun`) — which invokes the same binary twice on the
same `Input` — already covers header stability by construction. C6
adds one additional test (§6.1) asserting the header format across
every valid fixture's golden, so a future template regression that
smuggled a build timestamp / a UUID / a hostname into the header
fails at unit-test time, not at the golden-diff review surface.

- **Reproducibility.** Two different builds of the same binary at
  the same commit (same `-ldflags` invocation) must produce
  byte-identical output. The version string is the only text the
  header carries beyond the fixed `Code generated by gqlc ` prefix
  and the fixed `. DO NOT EDIT.\n\n` suffix (§5.2); pin the version
  string and the header is pinned.
- **No timestamp, no hostname, no UUID.** These are the standard
  determinism traps generators fall into (sqlc's `sqlc.yaml`-vs-CI
  divergence surfaced this repeatedly). The C6 spec pins the
  omission: the header carries `version` and nothing else.

### 2.3 What the C6 change means for the emitted module

The C5 emitted module stands byte-identical modulo the C6-tightened
header contract, which the emissions already satisfy. C6 is invisible
to the golden diff surface — every C0–C5 valid fixture's golden set
regenerates byte-identical. The nested-module compile fence (`just
test-codegen-fence`, C0 §7) proves the version-stamp extraction
compiles unchanged; the C6 header regex test (§6.1) proves the
format holds across the whole golden corpus.

## 4. Version-stamp hardening — the C6 delta

### 4.1 Source-of-truth: `internal/codegen/version.go`, `-ldflags -X`, no `runtime/debug`

The version string is a **package-level variable** in
`internal/codegen/version.go`, overridable at binary build time via
`-ldflags "-X github.com/areqag/gqlc/internal/codegen.version=vX.Y.Z"`:

```go
package codegen

// version is the stamp embedded in every generated file's header line
// (§5.2 of docs/specs/codegen-stage-c6.md). Default "dev"; release
// binaries override at build time:
//
//     go build -ldflags "-X github.com/areqag/gqlc/internal/codegen.version=$(git describe --tags)"
//
// The value is a package-level variable so the double-run determinism
// test (C0 §2.3) holds across arbitrary invocations of the same binary:
// two invocations of the same binary always see the same string.
var version = "dev"
```

- **`var`, not `const` — the `-ldflags -X` constraint.** Go's
  `-ldflags -X` overrides string *variables* at link time; a `const`
  cannot be overridden. `generate.go:22`'s current `const version =
  "dev"` is the C0 default that never fired the override; C6 lowers
  it to `var` so the release recipe actually takes effect. Same
  precedent as `runtime/debug.BuildInfo`'s many stdlib mirrors.
- **Unexported.** The version string is not part of the emitted
  package's public surface; consumers of `internal/codegen` (the CLI
  bead's future orchestration, the test suite) do not read it. The
  release recipe sets it via `-ldflags`; nothing else touches it.
- **New file, not `generate.go`.** The extraction is cosmetic and
  functional. Functional: `-ldflags "-X github.com/areqag/gqlc/internal/codegen.version=..."`
  is a build-time global-symbol substitution; putting the target
  symbol in its own file makes the substitution site trivially
  greppable (a future reader searching for "where does the version
  come from" hits `version.go` immediately). Cosmetic: `generate.go`
  is 2467 lines at C5; every extraction the polish stage can honour
  reduces the file's cognitive load without changing behaviour.

**Rejected — `runtime/debug.BuildInfo`.** The stdlib's
`runtime/debug.ReadBuildInfo()` returns module version data derived
from the caller's `go.mod`. Reading it at `generate.go` would let a
default-built `gqlc` binary carry a plausible `v0.0.0-<commit>`
version without a `-ldflags` override — attractive for the "dev
build stamp" DX. Rejected for three reasons:

1. **Runtime read violates the "no I/O, no runtime lookup" invariant
   (C0 §2.3).** The double-run test asserts that two invocations of
   the same binary on the same input produce byte-identical output.
   `runtime/debug.ReadBuildInfo()` returns build-time metadata, so
   the double-run invariant holds *within* a binary — but the
   determinism contract widens to "the header depends on where the
   binary was built" rather than "the header depends on the version
   the release recipe stamped". The narrower contract reads better.
2. **`BuildInfo` version strings are noisy.** A module built from a
   local checkout at a tagged commit shows
   `v0.0.0-<date>-<sha>+dirty` when uncommitted changes exist —
   noise the header would carry into every generated file. The
   `-ldflags` override is explicit; the release recipe sets it
   deliberately, and no local checkout accidentally embeds a
   `+dirty` marker.
3. **The `-ldflags` path is more universal.** Every reproducible-
   build recipe in Go (goreleaser, ko, plain Makefile) uses
   `-ldflags -X`. Betting on that path aligns gqlc with the standard
   Go release toolchain; betting on `BuildInfo` adds a second axis
   the CI recipe must consider.

**Rejected — a hardcoded literal.** A string like
`"v1.0.0-c6-spec"` frozen in the source treats every commit between
tags as the same version. Fine for `"dev"` (the default) — every
non-release build reads the same, and the `-ldflags` override
handles release builds — but not fine for the version stamp
proper. The `-ldflags` override at release time carries `git
describe --tags` output; the source default is the fallback for
the "someone built from a checkout without the release recipe"
case, and `"dev"` reads honestly as "not-a-release" without lying
about a version number.

### 4.2 Input-hash in the header? Rejected.

A candidate polish: extend the header line to include a hash over
the `Input` (schema + query set), so the generated file carries
change-detection metadata a downstream `git status` walk could use
to spot stale goldens:

```
// Code generated by gqlc dev (schema+queries sha256:abc123…). DO NOT EDIT.
```

Rejected. Three reasons:

1. **Goldens already do this.** The C0 harness is the change-
   detection surface: a fixture whose schema or query file changes
   diffs its `golden/` tree — that diff is the review surface, and
   `-update` regenerates. Hashing the input into the header
   duplicates the signal without adding a consumer.
2. **The hash would need a stable canonical form.** `Input.Schema`
   and `Input.Queries` are Go structs; hashing them requires a
   canonical serialisation (`gob`? `json.Marshal` with sorted map
   keys? a hand-rolled walker?). Every choice is a maintenance
   burden that would drift; every drift produces a header-diff
   without a corresponding source-diff, exactly the noise polish
   should reduce.
3. **The Go generated-file toolchain regex is the source of truth
   for "generated".** `^// Code generated .* DO NOT EDIT\.$`
   (Go 1.19+, verified against
   `pkg.go.dev/cmd/go/internal/generate`, 2026-07-11) matches the
   current header. Extending the header with parenthetical hash
   text still matches (`.*` is greedy), but every additional field
   the header carries is one more thing to defend against future
   drift. Minimal contract wins.

**Grill option: hash the `SourceText` of each query into a
per-source-file comment** (not the shared header). Rejected on the
same grounds: the per-source `.cypher.go` file already carries the
query text as a const (C1 §5.5); the const's diff surface is the
change-detection channel. A hash in a comment is signal
duplication without a consumer.

### 5.2 The header format — the C6 pin

Every emitted `.go` file (`db.go`, `models.go`, `querier.go`, every
`<name>.cypher.go`) begins with the header line + one blank line:

```
// Code generated by gqlc <version>. DO NOT EDIT.

package <derived>
```

Format contract (byte-level):

1. **Line 1 (fixed prefix):** `// Code generated by gqlc `
   (23 bytes: two slashes, one space, twenty-word chars). Fixed.
2. **Line 1 (version substitution):** `<version>` — the value of
   `codegen.version` (§4.1). No leading or trailing whitespace
   inside the substitution; a version override with embedded
   whitespace violates the contract and is a caller bug (the release
   recipe passes `git describe --tags` output, which by construction
   has no whitespace).
3. **Line 1 (fixed suffix):** `. DO NOT EDIT.` (14 bytes: period,
   space, five upper letters, space, three upper letters, space,
   four upper letters, period).
4. **Line 1 termination:** `\n` — a single LF (not CRLF, not
   trailing whitespace).
5. **Line 2:** empty (`\n` — a bare LF).
6. **Line 3:** the `package <derived>` clause.

The Go generated-file toolchain regex is
`^// Code generated .* DO NOT EDIT\.$` (verified against
`pkg.go.dev/cmd/go/internal/generate`, 2026-07-11): line 1 matches
by construction. `gofmt` / `goimports` / `staticcheck` / `revive`
respect the marker and skip the file's contents from their
generated-code-specific rules.

- **No build-tag lines.** The header carries no `//go:build` line.
  Every generated file compiles unconditionally in the user's
  module — build tags would fragment the surface.
- **No `// Package X ...` doc comment above the package clause.**
  The C1 spec omitted the doc comment; C6 pins the omission.
  Documentation about the emitted package's semantics belongs in the
  user's own doc.go / README, not injected by the generator.
- **The header is emitted by `codegen.header()`** (§4.2), one
  function, one call site per rendered file. Every renderer
  (`renderDB`, `renderQuerier`, `renderModels`, the per-source
  render walk) begins by writing `header()` and no other renderer
  emits a byte before it. The function's return value is a
  `string`; the callers `WriteString` or byte-append it.

**Grill option: parenthetical build metadata after the version.** A
release binary's header could carry
`// Code generated by gqlc v1.0.0 (built 2026-07-11). DO NOT EDIT.`
so consumers see the build date without a `git log` round-trip.
Rejected. Two reasons: (a) the version string via `git describe
--tags` already threads commit metadata (`v1.0.0-3-gabcdef`
disambiguates within a tag), and (b) the build date is a
determinism trap — two builds of the same commit at different
timestamps would diverge the header, violating the reproducibility
contract §2.2 pins. If a consumer wants the build date, `git show
$(gqlc --version)` gives it.

### 5.4 The `New` / `WithTx` seam and `SessionConfig` — the C6 audit

C0 §5.3 pinned the two constructors:

```go
func New(driver neo4j.DriverWithContext) *Queries {
    return &Queries{db: driverDB{driver: driver}}
}

func (q *Queries) WithTx(tx neo4j.ManagedTransaction) *Queries {
    return &Queries{db: txDB{tx: tx}}
}
```

C4 §5.3 wired the `driverDB.run` body's session construction with
`SessionConfig{AccessMode: access}`:

```go
session := d.driver.NewSession(ctx, neo4j.SessionConfig{AccessMode: access})
```

C6 pins the axes v1 locks in and the ones a future config effort
will widen.

#### 5.4.1 `SessionConfig` — fixed to `{AccessMode: access}` at v1

The generated `driverDB.run` constructs a fresh session per call
with `SessionConfig{AccessMode: access}` and no other field set.
The driver's `neo4j.SessionConfig` struct (verified against
`pkg.go.dev/github.com/neo4j/neo4j-go-driver/v5@v5.28.4/neo4j`,
2026-07-11) carries seven fields:

| Field                | Purpose                                    | v1 posture |
|----------------------|--------------------------------------------|------------|
| `AccessMode`         | read/write dispatch, cluster routing       | **set per-method** (C4) |
| `DatabaseName`       | multi-database routing (Neo4j 4+)          | zero → driver default |
| `ImpersonatedUser`   | auth-as-other-user (enterprise)            | zero → no impersonation |
| `Bookmarks`          | causal-consistency handoff                 | zero → no bookmark chain |
| `FetchSize`          | per-fetch record batch size                | zero → driver default (1000) |
| `BookmarkManager`    | bookmark-manager wiring                    | zero → nil (unmanaged) |
| `NotificationsMinSeverity` / `NotificationsDisabledClassifications` | notification filter | zero → driver default |

Every non-`AccessMode` field is unset — the driver applies its
defaults. This is the v1 policy: **owner-per-call session, driver
defaults for everything else**. The rationale (extending ADR 0010
D6's "no configuration in v1"):

- **The generated code makes no policy claim.** A user with an
  Aura deployment, a multi-database Neo4j instance, or an enterprise
  impersonation flow does not need generated code that knows about
  their topology; the driver does. Widening `SessionConfig` at v1
  would push the topology axis onto the generated surface and lock
  it into every user's `models.go` in a way a future config effort
  cannot un-do.
- **The right widening surface is the CLI, not the emitted code.** A
  future `gqlc.yaml` (D6 futures) may carry a `session_config`
  block that codegen reads and threads into a widened
  `driverDB.run`. That block is a config concern, not a codegen
  concern; C6 does not front-run it.

**Grill option — add `WithSessionConfig(cfg neo4j.SessionConfig) Option`
on `New()` at v1.** The option would let a user configure database
name, bookmarks, impersonation at construction time:

```go
q := codegen.New(driver, codegen.WithSessionConfig(neo4j.SessionConfig{
    DatabaseName: "movies",
    Bookmarks:    []string{lastCommittedBookmark},
}))
```

Rejected. Three reasons:

1. **The `Option` surface is a config surface in disguise.** ADR
   0010 D6 pinned "v1 has no configuration at all". Adding an
   `Option` type + variadic ctor arg puts config back on the table
   without the deliberate design D6 refused. A future config
   effort will have to un-decide the `Option` surface (or extend
   it), fragmenting the API story.
2. **The `Option` collides with the AccessMode-per-method dispatch.**
   `SessionConfig.AccessMode` is set per-call at C4 (`{AccessMode:
   access}`); a user-supplied `SessionConfig` with a different
   `AccessMode` would be silently overwritten. The interaction
   axis multiplies with every field the option carries — a real
   user with `Bookmarks: []string{...}` would expect them to
   thread through every call, but the C4 per-call session
   construction would need to merge the user's config with the
   per-call access mode. That merge is a policy decision (does the
   user's `AccessMode` win? do their `Bookmarks` win? both?) that
   v1 refuses to make.
3. **The current shape is not locked in.** The internal-API-freeze
   memory says gqlc has no consumer to protect on internal types.
   The emitted `New` shape *is* a locked-in public surface — but
   `New(driver neo4j.DriverWithContext) *Queries` is minimal, and
   future widening to `New(driver, opts ...Option) *Queries` is
   source-compatible (variadic args accept zero). The right time
   to add options is when config lands, not now.

**Grill option — expose `Queries.WithSession(cfg SessionConfig) *Queries`
as a chaining method.** Same rejections. The chaining method is
still an option surface disguised as fluent API; still front-runs
the config effort.

#### 5.4.2 `WithTx(nil)` — "trust the caller", no defensive nil-guard

The C0 spec pinned `WithTx(tx neo4j.ManagedTransaction) *Queries`
returning `&Queries{db: txDB{tx: tx}}`. If a caller passes a nil
`ManagedTransaction`, the returned `*Queries` wraps a `txDB` whose
`tx` field is nil. The first `q.run(...)` call (from any generated
method body) invokes `t.tx.Run(...)` on the nil interface — a runtime
nil-dereference panic at that call site.

C6 pins the "trust the caller" posture: **no defensive nil-guard in
`WithTx`, no error return.**

- **The signature stays `func (q *Queries) WithTx(tx neo4j.ManagedTransaction) *Queries`.**
  Not `(tx neo4j.ManagedTransaction) (*Queries, error)`. The house
  posture memory records: "trust internal, validate at boundary".
  The C6 read: `WithTx` is user-facing (external boundary), so the
  boundary-validation rule *would* apply — but the boundary in
  question is a nil-check on a driver-owned interface, and the
  boundary here is the driver's, not the generator's.
- **Nil-`ManagedTransaction` is a caller bug.** The type is
  `neo4j.ManagedTransaction` — the driver never returns nil from
  `ExecuteRead`/`ExecuteWrite`; a caller who passes nil constructed
  it manually or extracted it from a nil source. A defensive check
  would swallow the caller's own bug and delay the panic to the
  first `run` call, where the diagnostic ("nil deref in
  `txDB.run`") is *less* informative than the panic at the first
  method call ("nil deref calling method on `q.WithTx(nil)`"). Let
  the panic fire at the first-use site.
- **`WithTx` chaining stands.** A caller who chains
  `q.WithTx(tx1).WithTx(tx2)` gets a `*Queries` whose `db` is the
  txDB for `tx2`; the intermediate is discarded. This is the C0
  intended semantics — `WithTx` returns a *new* `*Queries` because
  the driver's transaction lifetime is scoped to the callback the
  caller passed to `ExecuteWrite`, so a caller mid-callback wanting
  to re-scope to a different tx re-calls `WithTx`. Chain-and-discard
  is legal, idempotent (each call is a fresh wrap), and cheap.
- **`WithTx` interaction with the run indirection.** The `*Queries`
  returned from `WithTx` has `db` typed as `txDB`; every subsequent
  method call routes through `txDB.run`, which ignores the
  `AccessMode` argument (C0 §5.3, C4 §5.3 unchanged). The caller
  established the transaction's access mode by calling
  `ExecuteRead` vs `ExecuteWrite`; re-passing `AccessMode` through
  the seam would be a policy decision the transaction owner has
  already made.

**Grill option — validate at the boundary.** Change `WithTx` to:

```go
func (q *Queries) WithTx(tx neo4j.ManagedTransaction) (*Queries, error) {
    if tx == nil {
        return nil, errors.New("gqlc: WithTx: nil ManagedTransaction")
    }
    return &Queries{db: txDB{tx: tx}}, nil
}
```

Rejected. Three reasons:

1. **Signature churn against sqlc precedent.** sqlc's `WithTx(tx *sql.Tx) *Queries`
   returns a `*Queries`, not `(*Queries, error)`. Deviating from
   the well-known shape for a defensive check on a caller bug is
   friction with no user-facing benefit.
2. **`errors.New` at the boundary would need a sentinel.** House
   posture is closed sentinel sets; a new `ErrNilTransaction`
   sentinel to defend against the caller bug is over-fit for the
   fault surface. The first-use panic already discriminates the
   fault ("nil deref in `t.tx.Run`") with the same information.
3. **The nil-guard would migrate the fault, not remove it.** A
   caller who did `q, _ := q0.WithTx(nil); q.CreatePerson(ctx, ...)`
   would still panic (nil `*Queries` deref) at the first method
   call. The nil-guard just moves the panic upstream by one call
   frame. Panicking at the first method call is the honest posture.

#### 5.4.3 Cluster routing — documented but not enforced by any test

`AccessModeRead` routes to followers (in an Aura or clustered
deployment); `AccessModeWrite` routes to the leader. This is the
driver's routing policy; the generated code passes the axis and
takes no further action.

C4 §5.4 documented the axis; no test enforces it because no test
harness inside the nested golden module runs against a live
clustered deployment. C6 pins the posture: **v1 documents cluster
routing but adds no test.** Rationale:

- **The test would require a live cluster.** A test that spins up
  a two-node Aura-alike, runs a `:one` read + a `:one` write, and
  asserts the read hit the follower would require testcontainers,
  a Neo4j Enterprise image, and a topology configuration —
  infrastructure the golden harness does not carry. The unit-test
  layer covers the value passed to `SessionConfig.AccessMode`; the
  driver's routing is out of scope.
- **The driver's routing is the tested surface.** neo4j-go-driver
  v5's own test suite covers cluster routing exhaustively. A gqlc
  test would re-test the driver's contract, not gqlc's.
- **A future integration test suite** (post-v1, outside the golden
  harness) may cover the axis end-to-end. C6 files it as a note
  in the out-of-scope table (§10) rather than opening a bead.

### 6.1 New test — `TestGeneratedHeaderFormat`

A new test in `internal/codegen/codegen_test.go` asserts the header
line format against every valid fixture's generated `.go` files:

```go
func TestGeneratedHeaderFormat(t *testing.T) {
    // Regex: fixed prefix, one non-whitespace version token, fixed suffix.
    headerRe := regexp.MustCompile(`^// Code generated by gqlc \S+\. DO NOT EDIT\.\n\n`)
    // ... walk test/data/codegen/valid/*/golden/*.go, assert every file's
    // first 60 bytes (approximately, covers longest expected header) matches.
}
```

- **Regex specifics.** The `\S+` between the fixed prefix and
  suffix matches the version token — one-or-more non-whitespace
  characters — so a hypothetical `-ldflags -X` release version
  (`v1.0.0`, `v1.0.0-3-gabcdef`, `v0.0.0-20260711-abcdef`) all
  match, and a pathological override with whitespace (`v1.0 dev`)
  fails at test time.
- **`\n\n` termination.** The header ends with the LF ending line 1
  and the LF-only blank line 2. A future template regression that
  inserted a build-tag line or a doc comment between the header
  and the package clause would fail this test.
- **Walk every valid fixture.** The test iterates every
  `test/data/codegen/valid/<fixture>/golden/*.go`, so any C0–C5
  fixture whose golden set was regenerated at an earlier stage is
  covered by the current fence. New C7+ (post-v1) fixtures inherit
  the coverage automatically.

**Alternative — extract the header regex as a package-level
`internal/codegen` var.** Rejected: the regex is a test-only
concern; exposing it as a package-level `HeaderRegex` var invites
consumers to depend on it and would freeze the format under a
public contract. The test-local regex serves the invariant without
the surface cost.

## 7. Sentinel set delta — the C6 view

C5's fifteen sentinels stand at C6 with one rename and zero
additions or retirements. C6 renames `ErrOutOfC5Scope` →
`ErrOutOfC6Scope` for the same reason C5 renamed from C4 (per-stage
rename discipline, §9 of C5 spec defence extends). C6 adds no new
sentinel: version-stamp hardening is a documentation pin, session-
config polish is a documentation pin, `gqlc-0aa` re-scope closes an
external bead — none of these introduce a new rejection axis.

**Renamed sentinel:**

```go
// ErrOutOfC6Scope is returned when a C6-admissible input carries a
// construct C6 does not project: a non-property parameter (post-v1;
// whole-node / whole-edge / scalar-literal / list / unknown / bare-
// temporal-expression parameter is still out of scope), or a query
// text carrying a raw-string-hostile backtick. Category-grained per
// C0's precedent. Renamed from ErrOutOfC5Scope at C6; no scope-
// widening at C6 (polish stage, no new capability admitted).
var ErrOutOfC6Scope = errors.New("out of C6 scope")
```

**Retired at C6:** `ErrOutOfC5Scope` — the constant is dropped from
the package, the fixture renames to `out_of_c6_scope_*`, and the
`sentinelByName` map's entry renames. The retirement is a clean cut:
no `//nolint:staticcheck` for a lingering alias, no deprecation
window.

**Naming defence — `ErrOutOfC6Scope`, per-stage rename on its own
merits.** C5's §9 defence extends verbatim. **A grill open question:
is the rename discipline mechanical or scoped to real behaviour
changes?** C6 is the polish stage — the sentinel's *catchment* did
not change (a non-property parameter is still post-v1; a raw-string-
hostile backtick is still post-v1). C4 → C5 renamed because
edgeUnion retired from the catchment; C5 → C6 has no capability-
retirement to point at. Two answers considered:

- **Mechanical answer (picked).** The per-stage rename is a
  discipline, not a catchment-driven rename. Every stage boundary
  produces a new binary; a caller who branched on
  `ErrOutOfC5Scope` should re-inspect their code at the C6 upgrade
  regardless of whether the sentinel's catchment moved. The rename
  is a forcing function for that re-inspection. Same rationale C4
  and C5 documented; C6 extends it verbatim.
- **Behaviour-scoped answer (rejected).** If the catchment did not
  move, freeze the name at `ErrOutOfC5Scope` and only rename on
  catchment change. Rejected because the discipline's *predictability*
  matters more than the rename's *justification*: every stage has
  the same rename; readers know what to expect. A "sometimes-
  rename" rule would need a defence at every stage boundary about
  whether the catchment moved.

**Rejected — a new sentinel for "version stamp malformed".** A
hypothetical caller-supplied version override with whitespace
(`-ldflags "-X ...version=v1.0 dev"`) would violate the §5.2
format. Rejected as sentinel-worthy: the `-ldflags` invocation is a
build-recipe concern, and `TestGeneratedHeaderFormat` (§6.1)
catches the drift at test time, not at generation time. The
generator writes the string it is given.

## 8. `gqlc-0aa` re-scope — close-outright

The bead `gqlc-0aa` (Driver wiring: runtime layer for generated
repositories) was filed pre-C0, when a driver-wiring runtime
package seemed possible. ADR 0010 D4 killed the runtime package
(generated code imports the driver directly; there is no gqlc
runtime layer for the user's module to import). D4 flagged
`gqlc-0aa` for re-scope at C6.

**C6 decision: close `gqlc-0aa` outright.** No session/tx helper
package needed. The reasoning:

1. **Session opening is inline in `driverDB.run`** (C4 §5.3).
   `d.driver.NewSession(ctx, neo4j.SessionConfig{AccessMode:
   access})` is the one line; extracting a helper package
   `codegen/runtime.OpenSession(driver, access)` would fragment
   the seam across two files and buy nothing (the one line reads
   at the call site).
2. **Transaction chaining is `WithTx`** (C0 §5.3, C6 §5.4.2).
   The generated `Queries.WithTx(neo4j.ManagedTransaction) *Queries`
   is one method, six lines including docstring. No runtime
   package needed to wrap what the emitted code owns directly.
3. **Access-mode dispatch is per-method** (C4 §5.3). Every
   method's `run` call takes `neo4j.AccessModeRead` or
   `neo4j.AccessModeWrite` per its statement kind; the dispatch is
   a one-line template expansion in the per-source file, not a
   runtime lookup.
4. **The `ManagedTransaction` path is in `txDB.run`** (C0 §5.3).
   Three lines: run the query, error-check, return records. No
   runtime helper.
5. **Cluster routing is the driver's** (C6 §5.4.3). The generated
   code passes the access mode; the driver routes. No runtime
   helper.

Every capability the driver-wiring bead would have owned turns out
to have been subsumed by C0–C5's per-file emissions:

| gqlc-0aa's original scope        | Delivered by                | Runtime package? |
|----------------------------------|------------------------------|------------------|
| Session opening                  | `driverDB.run` (C4 §5.3)    | No               |
| Query dispatch                   | Per-method body (C1–C4)     | No               |
| Result decoding into typed Go    | Entity decode helpers + Row assembly (C2–C5) | No |
| Transaction composition          | `WithTx` (C0 §5.3)          | No               |
| Cluster routing                  | Driver-native (C4, C6 §5.4.3) | No             |

**Close-out mechanics** (owner action, post-spec-merge):

- `bd close gqlc-0aa` with reason "closed by C6: ADR 0010 D4's
  no-runtime-package decision subsumed every gqlc-0aa capability;
  no session/tx helper package needed. See docs/specs/codegen-
  stage-c6.md §8."
- The GH mirror (`#150`) closes with the same reason. Per the
  PR-hygiene memory, `gqlc-0aa` is a direct 1:1 bd↔GH issue, so
  the C6 code PR body writes `Closes #150`.

**Grill option — keep `gqlc-0aa` as a follow-up for post-v1
integration testing.** A post-v1 "run against a live neo4j"
integration test suite (§5.4.3) may want a small helper package to
wire fixtures. Rejected. Two reasons:

1. **The bead's title is "Driver wiring: runtime layer".** An
   integration-test-fixture bead is a different concern; filing
   it under `gqlc-0aa` would rename the bead's scope silently.
   File a fresh bead when the integration suite is planned.
2. **YAGNI applies.** No integration test suite is on the current
   plan; keeping the bead open as a placeholder for hypothetical
   future work invites the "why is this open?" thrash beads
   should avoid.

**Grill option — re-scope `gqlc-0aa` to "documentation: driver
wiring how-to".** A README section explaining "how to construct
`*Queries`, use `WithTx`, handle errors". Rejected: the emitted
package's docstrings + the ADR 0010 D2 description already cover
this. A separate documentation bead would duplicate.
